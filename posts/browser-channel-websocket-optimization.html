问题背景

旧架构的痛点

场景：用户在聊天界面发送了一条复杂指令，Agent 开始执行多个并行工具（如同时搜索多个文件）。用户突然想取消这个操作，点击「停止」按钮，但...
  • 请求发送到了服务器，但 Agent 正在 asyncio.gather() 中等待工具完成
  • 由于 asyncio.gather() 会阻塞直到所有任务完成，无法在执行过程中检查取消状态
  • 用户只能等待所有工具执行完毕，才能看到「已停止」的效果
  • 更糟糕的是：某些工具（如网络请求）可能需要等待超时才能返回

解决方案

1. AgentLoop：取消轮询机制

核心思路：不再使用 asyncio.gather() 阻塞等待，而是采用「等待 + 轮询取消状态」的方式。

新增 _gather_awaitables_with_cancel_poll 方法：
  • 使用 asyncio.wait() 替代 asyncio.gather()
  • 每次有任务完成时，轮询检查用户是否取消了请求
  • 如果检测到取消，立即取消所有待完成的任务
async def _gather_awaitables_with_cancel_poll(
    self,
    awaitables: list,
    session_key: str | None,
    *,
    interval: float = 1.0,
    return_exceptions: bool = False,
) -> list:
    """并行等待多个 awaitable，期间轮询用户取消；避免 asyncio.gather 长时间阻塞无法响应停止。"""
    if not awaitables:
        return []
    tasks = [asyncio.ensure_future(a) for a in awaitables]
    pending = set(tasks)
    try:
        while pending:
            done, pending = await asyncio.wait(
                pending, timeout=interval, return_when=asyncio.FIRST_COMPLETED
            )
            # 每次有任务完成时，检查是否被取消
            await self._check_cancelled(session_key)
        # ... 返回结果
    except BaseException:
        # 任何异常都取消所有任务
        for t in tasks:
            if not t.done():
                t.cancel()
        await asyncio.gather(*tasks, return_exceptions=True)
        raise

技术细节

为什么选择轮询而非回调？

常见的解决方案是让每个工具注册一个「取消回调」，但这需要修改所有工具的实现。轮询方案的优点：

  • 侵入性小：不需要修改现有工具代码
  • 可配置：可以调整轮询间隔 interval
  • 统一处理：取消逻辑集中在一处

为什么是 1 秒轮询间隔？

interval 设置为 1.0 秒是一个平衡：

  • 足够快：用户点击停止后，最多 1 秒就会响应
  • 开销小：不会对性能造成显著影响
  • 兼容性好：适用于大多数工具的执行时间

部分回复的保留

取消时，Agent 可能已经生成了部分回复。通过 sessions.get_messages() 获取已保存的消息，确保用户可以看到已经完成的部分内容。
